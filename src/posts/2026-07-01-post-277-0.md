### 為何要自建 DeepSeek API 網關？

我之前就發現，直接把 DeepSeek API 金鑰分給各個團隊，管理太麻煩，萬一洩露更是麻煩大了。自己搭網關，能把 DeepSeek 的『主金鑰』穩穩藏在服務器，對外只放驗證過的接口。另外，網關還有這些好處：

1.  **統一入口**：所有請求都經網關，管理方便。
2.  **權限細化**：給不同用戶或應用設訪問權限，甚至限制每秒請求數（Rate Limiting）。
3.  **用量統計**：精確記錄每次調用，包括用戶、應用、用了多少 tokens，內部計費有數據。
4.  **安全強化**：集中處理身份驗證、日誌記錄和潛在攻擊防禦。

### 準備工作：你需要甚麼？

動手前，得準備一台能跑服務的伺服器（雲機、辦公室迷你機都行），對 Linux 基礎、Nginx 或 Web 框架（像 Node.js 的 Express、Python 的 Flask）有點概念。這篇文主要用 Node.js 加 Nginx 演示，它輕量好部署。記住，Node.js 最好 18.x+，Nginx 至少 1.22.0。

### 第一步：架設基礎 API 轉發服務

第一步，就是搞個服務來接收客戶端請求，再轉發給 DeepSeek API。可以用 Nginx 做反向代理，也能寫個 Node.js 服務。考慮到後續更多邏輯處理，我更建議用 Node.js 來搞。

建立一個 `gateway.js` 檔案：

```javascript
const express = require('express');
const axios = require('axios');
const app = express();
const PORT = process.env.PORT || 3000;

app.use(express.json());

app.all('/v1/*', async (req, res) => {
  try {
    // 這裡只是簡單轉發，後續步驟會加入權限驗證
    const deepseekApiKey = process.env.DEEPSEEK_API_KEY; // 從環境變量獲取主金鑰
    const targetUrl = `https://api.deepseek.com${req.originalUrl}`;
    const response = await axios({
      method: req.method,
      url: targetUrl,
      headers: {
        'Content-Type': req.headers['content-type'] || 'application/json',
        'Authorization': `Bearer ${deepseekApiKey}`,
        'Accept': req.headers['accept'] || 'application/json'
      },
      data: req.body
    });
    res.status(response.status).send(response.data);
  } catch (error) {
    console.error('DeepSeek API 轉發錯誤:', error.message);
    res.status(error.response?.status || 500).send(error.response?.data || '內部伺服器錯誤');
  }
});

app.listen(PORT, () => {
  console.log(`API Gateway 服務已在埠 ${PORT} 啟動`);
});
```

裝好 `npm install express axios`，再跑 `node gateway.js`，你的轉發服務就起來了。DeepSeek API 金鑰咋管？看這篇：[DeepSeek API 金鑰管理策略](/posts/deepseek-api-key-and-limits/)。

### 第二步：實作靈活的權限驗證機制

這步是網關的關鍵！不能讓誰都能用咱們的網關吧？最保險就是給每個內部應用或用戶發個專用內部 API 金鑰，網關來驗證。

修改 `gateway.js`，加入權限驗證邏輯（以下是簡化範例，實際生產環境會更複雜，可能涉及數據庫）：

```javascript
// ... (前面的引入和 app.use(express.json()) 不變)

// 假設我們有一個用戶金鑰列表，真實場景會從數據庫讀取
const VALID_INTERNAL_API_KEYS = {
  'proj-finance-key-123': { name: '財務報表專案', usageLimit: 10000 },
  'dept-hr-key-456': { name: '人力資源聊天機械人', usageLimit: 5000 }
};

app.all('/v1/*', async (req, res) => {
  const internalApiKey = req.headers['x-internal-api-key']; // 假設客戶端會傳遞這個自定義頭

  if (!internalApiKey || !VALID_INTERNAL_API_KEYS[internalApiKey]) {
    return res.status(401).send('未經授權：無效或缺失的內部 API 金鑰');
  }

  // 此處可以加入更多邏輯，如檢查 `usageLimit`，實現限速等
  const userInfo = VALID_INTERNAL_API_KEYS[internalApiKey];
  console.log(`請求來自：${userInfo.name} (${internalApiKey})`);

  try {
    const deepseekApiKey = process.env.DEEPSEEK_API_KEY;
    const targetUrl = `https://api.deepseek.com${req.originalUrl}`;
    const response = await axios({
      method: req.method,
      url: targetUrl,
      headers: {
        'Content-Type': req.headers['content-type'] || 'application/json',
        'Authorization': `Bearer ${deepseekApiKey}`,
        'Accept': req.headers['accept'] || 'application/json'
      },
      data: req.body
    });
    res.status(response.status).send(response.data);
  } catch (error) {
    console.error('DeepSeek API 轉發錯誤:', error.message);
    res.status(error.response?.status || 500).send(error.response?.data || '內部伺服器錯誤');
  }
});
// ... (後面的 app.listen 不變)
```

客戶端調用網關，請求頭得帶 `X-Internal-API-Key: your-internal-key`。網關先驗證內部金鑰，通過了才拿主金鑰跟 DeepSeek 服務器通信。這樣安全性蹭蹭漲！

### 第三步：追蹤與計費統計

權限搞定後，每次請求成功就能記錄數據，方便分析和算賬。DeepSeek API 返回的 `usage` 字段裡有 `prompt_tokens` 和 `completion_tokens`，這就是咱們計費的憑據。

在成功響應後，加入日誌記錄：

```javascript
// ... (前面的代碼不變)

app.all('/v1/*', async (req, res) => {
  const internalApiKey = req.headers['x-internal-api-key'];
  // ... (權限驗證邏輯)

  try {
    // ... (DeepSeek API 請求邏輯)
    
    const response = await axios({
      // ... (請求配置)
    });

    // 記錄請求詳情
    const usageData = response.data.usage;
    if (usageData) {
      console.log(`[${new Date().toISOString()}] 用戶/專案: ${userInfo.name} (${internalApiKey}) | 調用 DeepSeek 模型: ${req.body.model || '未知'} | 輸入 Token: ${usageData.prompt_tokens} | 輸出 Token: ${usageData.completion_tokens} | 總 Token: ${usageData.total_tokens}`);
      // 在生產環境中，這裡會將數據寫入數據庫或日誌系統
    } else {
      console.log(`[${new Date().toISOString()}] 用戶/專案: ${userInfo.name} (${internalApiKey}) | 調用 DeepSeek API 成功，但未獲取到使用量數據`);
    }

    res.status(response.status).send(response.data);
  } catch (error) {
    // ... (錯誤處理)
  }
});
// ... (後面的 app.listen 不變)
```

這些日誌數據能存文件、Elasticsearch 或時序數據庫，再搭個 Grafana，DeepSeek API 怎麼用的，清清楚楚！想優化提示詞，這些用量數據超級重要。更多玩法看這篇：[DeepSeek 提示詞優化入門](/posts/deepseek-prompt-basics/)。

### 常見問題

**Q1：能直接用現成 API 網關服務嗎？**

A1：當然可以啊！像 Kong、Apigee 或 AWS API Gateway 功能賊強大、穩定。但中小企自己搭輕量網關，成本低、控制強，初期絕對夠用。這篇文主要就是讓你懂原理，自己搞定。

**Q2：DeepSeek API 老是限速咋辦？**

A2：DeepSeek 自己有限速。網關這層，能給每個 `internalApiKey` 設訪問頻率限制，比如用 `express-rate-limit`。如果 DeepSeek 回 429，網關還能加自動重試，提升用戶體驗。具體看：[DeepSeek API 重試策略實戰](/posts/deepseek-api-retry-guide/)。

**Q3：主金鑰安全嗎？**

A3：安全第一！DeepSeek API 主金鑰必須設成環境變量（`process.env.DEEPSEEK_API_KEY`），絕不能寫死在代碼裡！部署時確保服務器環境安全，限制變量訪問。用 Docker 或 Kubernetes，它們的秘密管理功能超好用。
