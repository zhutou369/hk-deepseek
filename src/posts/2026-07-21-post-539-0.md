## 設計你的「醒目」重試策略

搞重試機制，唔係求其亂咁試，而係要「醒目」地做！一個掂嘅策略，要諗埋服務器會唔會超載、網絡會唔會延遲，同埋錯咗啲乜嘢。下面係我個人推薦嘅三步走策略：

### 步驟一：基礎延遲與最大重試次數設定

最簡單嘅重試就係等等再試。不過呢，齋等固定時間其實唔係最抵玩，而且重試都要有上限㗎嘛，唔可以無限次咁嚟！

所以，我哋要先諗清楚兩樣嘢：
1.  **最大重試次數 (max_retries)**：通常設 3 到 5 次就夠應付大多數短暫問題啦。
2.  **初始延遲 (initial_delay)**：第一次重試前等幾耐？0.5 到 1 秒開始會比較穩陣。

睇下呢個基本嘅 Python 骨架：

```python
import time
import requests # 假設你用 requests 或者 deepseek-python-sdk 底層係 requests/httpx

def call_deepseek_api_simple(prompt, max_retries=3, initial_delay=1):
    for i in range(max_retries + 1):
        try:
            # 實際 DeepSeek API 調用
            # response = deepseek.chat.completions.create(...)
            # 這裡用一個模擬的請求
            print(f"嘗試調用 DeepSeek API (第 {i+1} 次)")
            if i < 2: # 模擬前兩次超時
                raise requests.exceptions.Timeout("模擬超時")
            
            # 成功處理
            print("API 調用成功！")
            return {"result": "成功獲取回應"}
        except requests.exceptions.Timeout as e:
            if i < max_retries:
                print(f"API 調用超時，將在 {initial_delay} 秒後重試...")
                time.sleep(initial_delay)
            else:
                print(f"API 調用 {max_retries} 次後仍然超時。")
                raise e
        except Exception as e: # 其他非超時錯誤
            print(f"發生其他錯誤: {e}")
            raise e
    return None

# 簡單測試
```

### 步驟二：引入指數退避與隨機抖動

齋用固定延遲會有個麻煩：萬一大堆請求同一時間仆街，又喺同一時間重試，就好容易搞出「雷鳴群效應」（Thundering Herd Problem），反而塞爆服務器，搞到更多超時。為咗避開呢個雷，我哋要識得用兩個法寶：

1.  **指數退避 (Exponential Backoff)**：每次等嘅時間都比上次長一倍，好似 1s, 2s, 4s, 8s 咁。咁做可以慢慢拉開重試嘅時間，等服務器可以抖下氣。
2.  **隨機抖動 (Jitter)**：喺指數退避嘅基礎上，再加少少隨機時間，咁就唔會每次都準時重試，可以再分散啲請求。

```python
import time
import random
import requests # 假設你用 requests 或者 deepseek-python-sdk 底層係 requests/httpx

def call_deepseek_api_with_backoff(prompt, max_retries=5, initial_delay=1, max_delay=60):
    for i in range(max_retries + 1):
        try:
            print(f"嘗試調用 DeepSeek API (第 {i+1} 次)")
            if i < 3: # 模擬前三次超時
                raise requests.exceptions.Timeout("模擬超時")
            
            print("API 調用成功！")
            return {"result": "成功獲取回應"}
        except requests.exceptions.Timeout as e:
            if i < max_retries:
                # 計算指數退避延遲，並加入隨機抖動
                current_delay = min(max_delay, initial_delay * (2 ** i))
                jitter = random.uniform(0, current_delay * 0.2) # 20% 抖動
                sleep_time = current_delay + jitter
                print(f"API 調用超時，將在 {sleep_time:.2f} 秒後重試...")
                time.sleep(sleep_time)
            else:
                print(f"API 調用 {max_retries} 次後仍然超時。")
                raise e
        except Exception as e:
            print(f"發生其他錯誤: {e}")
            raise e
    return None

# 簡單測試
```

### 步驟三：錯誤類型判斷與條件式重試

唔係所有錯都值得重試㗎！好似如果你個請求參數都寫錯咗（例如 HTTP 400 Bad Request），你試一百次都係錯㗎嘛。我哋只應該針對啲「暫時性」、「有得救」嘅錯誤嚟重試，好似：
*   **Timeout (超時)**：`requests.exceptions.Timeout`, `httpx.TimeoutException` 呢類。
*   **Connection Error (連接錯誤)**：`requests.exceptions.ConnectionError` 都係。
*   **Server Error (伺服器錯誤)**：所有 HTTP 5xx 狀態碼，例如 500, 502, 503, 504 呢啲，通常都係伺服器嗰邊暫時性出問題。

講返 DeepSeek API，佢多數會拋出 `deepseek.APIError` 異常，入面有埋 HTTP 狀態碼。我哋就係靠呢個狀態碼嚟判斷點處理。記住呀，DeepSeek API Key 設定好重要，唔識點搞可以睇下呢篇文：[DeepSeek API Key 與限制完全攻略](/posts/deepseek-api-key-and-limits/)。

```python
import time
import random
import requests
from deepseek import Deepseek, APIError, APIConnectionError, APIStatusError # 假設 deepseek SDK 有這些錯誤類型

# 初始化 DeepSeek 客戶端，記得設定好你的 API Key
# client = Deepseek(api_key="YOUR_DEEPSEEK_API_KEY_HERE")

def is_retryable_error(e):
    # 檢查是否為可重試的錯誤類型
    if isinstance(e, (requests.exceptions.Timeout, APIConnectionError)):
        return True
    if isinstance(e, APIStatusError) and e.status_code >= 500:
        # 5xx 錯誤通常是伺服器端暫時性問題
        return True
    return False

def call_deepseek_api_robust(prompt_content, max_retries=5, initial_delay=1, max_delay=60):
    client = Deepseek(api_key="YOUR_DEEPSEEK_API_KEY_HERE") # 在實際應用中，client 應該作為參數傳入或全局變量
    for i in range(max_retries + 1):
        try:
            print(f"嘗試調用 DeepSeek API (第 {i+1} 次)，內容：'{prompt_content[:20]}...'")
            response = client.chat.completions.create(
                model="deepseek-chat",
                messages=[
                    {"role": "user", "content": prompt_content},
                ],
                stream=False,
                timeout=20 # 設定 DeepSeek API 客戶端的超時時間
            )
            print("API 調用成功！")
            return response
        except (requests.exceptions.Timeout, APIConnectionError, APIStatusError) as e:
            if is_retryable_error(e) and i < max_retries:
                current_delay = min(max_delay, initial_delay * (2 ** i))
                jitter = random.uniform(0, current_delay * 0.2)
                sleep_time = current_delay + jitter
                print(f"API 調用遇到可重試錯誤: {e.__class__.__name__}。將在 {sleep_time:.2f} 秒後重試...")
                time.sleep(sleep_time)
            else:
                print(f"API 調用 {max_retries} 次後仍然失敗或遇到不可重試錯誤: {e}")
                raise e
        except Exception as e:
            print(f"發生意外錯誤: {e}")
            raise e
    return None
```

## 實戰代碼：一個完善的 DeepSeek API 重試函數

頭先講嗰啲步驟係逐個拆開嚟講，但實際開發時，我哋會將佢哋包裝埋一個實用函數度。如果你嫌自己寫咁多邏輯麻煩，Python 界有個超正嘅庫叫 `tenacity`，佢就係專為重試而生，勁好用㗎！

例如，用 `tenacity` 實現指數退避同抖動：

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from deepseek import Deepseek, APIError, APIConnectionError, APIStatusError
import requests.exceptions

# 記得替換成你的實際 API Key，或者從環境變量讀取
DEEPSEEK_API_KEY = "YOUR_DEEPSEEK_API_KEY_HERE"

# 定義一個函數來判斷哪些 DeepSeek 錯誤是可重試的
def is_deepseek_retryable_error(exception):
    if isinstance(exception, (requests.exceptions.Timeout, APIConnectionError)):
        return True
    if isinstance(exception, APIStatusError) and exception.status_code >= 500:
        return True
    return False

@retry(
    wait=wait_exponential(multiplier=1, min=1, max=60), # 指數退避，最小1秒，最大60秒
    stop=stop_after_attempt(5), # 最多重試5次
    retry=retry_if_exception_type((requests.exceptions.Timeout, APIConnectionError, APIStatusError)) & retry_if_exception_type(is_deepseek_retryable_error),
    reraise=True # 如果所有重試都失敗，重新拋出最後一個異常
)
def call_deepseek_with_tenacity(prompt_content: str):
    """
    使用 tenacity 進行重試的 DeepSeek API 調用函數。
    """
    client = Deepseek(api_key=DEEPSEEK_API_KEY) # 這裡每次都初始化 client，實際中建議傳入或用單例模式
    print(f"嘗試調用 DeepSeek API，內容：'{prompt_content[:20]}...'")
    response = client.chat.completions.create(
        model="deepseek-chat",
        messages=[
            {"role": "user", "content": prompt_content},
        ],
        stream=False,
        timeout=30 # 設定 DeepSeek API 客戶端的超時時間
    )
    print("API 調用成功！")
    return response
```

## 常見問題

### Q1: 重試要試幾多次先夠？
A: 普通情況下，試 3 到 5 次通常都夠啦。如果係啲超重要、又唔急住要即時反應嘅後台嘢，最多可以加到 10 次。但唔好設太多次，唔係你個程式會等太耐，又浪費資源。

### Q2: 指數退避會唔會等太耐？
A: 指數退避入面有個超重要參數叫 `max_delay`。你可以按你實際需求嚟較，好似我哋上面設 60 秒咁。如果 API 幾分鐘都未好返，咁可能就唔係一時三刻嘅小問題，而係出大鑊啦，繼續盲目重試都無乜用。所以，較個啱啱好嘅 `max_delay`，就可以平衡到個程式嘅恢復能力同反應速度。

### Q3: 除咗超時，仲有咩錯要重試？
A: 除咗超時，所有 HTTP 5xx 錯誤（好似 500, 502, 503, 504 呢啲）通常都係重試嘅好目標，因為多數係服務器嗰邊暫時性抽風。連接錯誤（`ConnectionError`）都係。DeepSeek API 嘅話，你可能撞到 `deepseek.APIConnectionError` 或 `deepseek.APIStatusError`，記得對症下藥。有時 DeepSeek 會限制你發送請求嘅速度，彈個 429 Too Many Requests 出嚟，呢個都係要重試嘅常見情況，通常會睇 `Retry-After` 個頭部嚟精確延遲。想寫出更強勁嘅 DeepSeek 提示詞？唔使問阿貴，快啲去睇下呢篇：[DeepSeek Prompt 基本原理與實踐](/posts/deepseek-prompt-basics/)。
