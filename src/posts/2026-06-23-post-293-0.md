## 自建DeepSeek API中轉站的技術優勢

利用Cloudflare搭建中轉站，唔單止係為咗解決網絡問題，更係一種技術優化同成本控制嘅策略。

### 改善連接穩定性與速度

通過Cloudflare Workers，您嘅API請求會先到達Cloudflare全球最近嘅邊緣節點。由呢度，Cloudflare會利用其優化嘅骨幹網絡將請求轉發到DeepSeek API伺服器，再將響應傳回用戶。呢個過程有效繞過咗公共互聯網上可能存在嘅擁堵點同地域限制，顯著提高連接穩定性同響應速度。對於香港同大中華區嘅用戶嚟講，效果尤其明顯。

### 提升數據安全性

將DeepSeek API Key直接暴露喺客戶端（例如網頁前端或手機App）係一個極大嘅安全隱患。一旦API Key洩露，可能導致服務被濫用，產生不必要嘅費用。透過中轉站，API Key只會存儲喺Cloudflare Worker嘅環境變量中，由Worker負責喺伺服器端添加至請求頭，客戶端永遠唔會接觸到。呢大大提升咗API Key嘅安全性。

### 靈活的流量管理與負載均衡

Cloudflare Workers本身就具備強大嘅負載均衡能力。所有請求都會自動分發到最近且負載最低嘅Worker實例。未來您仲可以進一步配置更複雜嘅路由規則、限速策略，甚至多個DeepSeek API Key嘅輪換使用，以應對高併發流量或特定業務需求。

### 成本效益

Cloudflare Workers提供慷慨嘅免費額度，每月數百萬次請求同數十萬GB嘅數據傳輸通常足夠中小企同個人開發者使用。即使超出免費額度，付費方案嘅價格亦相當合理，遠低於自己租用雲伺服器並進行維護嘅成本。善用免費資源，係香港中小企數字轉型嘅重要一環。

## 逐步教學：利用Cloudflare Workers搭建中轉站

以下係一步步教您點樣利用Cloudflare Workers搭建DeepSeek API中轉站嘅詳細流程。

### 前提準備

1.  **Cloudflare 賬戶：** 確保您已經擁有一個Cloudflare賬戶並已登入。
2.  **DeepSeek API Key：** 喺DeepSeek AI官網註冊並生成您嘅API Key。請務必妥善保管。
3.  **（可選）自定義域名：** 如果您希望中轉站使用自己嘅域名（例如`api.yourdomain.com`），請確保該域名已接入Cloudflare。

### 第一步：創建Cloudflare Worker

1.  登入Cloudflare儀表板。
2.  喺左側導航欄，選擇 **Workers & Pages**。
3.  點擊「**創建應用程式**」按鈕，然後選擇「**創建Worker**」。
4.  為您嘅Worker服務命名（例如：`deepseek-proxy`），然後點擊「**部署**」。
5.  部署完成後，點擊「**編輯代碼**」。

### 第二步：編寫Worker腳本

呢個Worker腳本嘅核心任務係攔截所有傳入請求，添加DeepSeek API Key，然後將請求轉發到DeepSeek嘅官方API端點，最後將響應返回給客戶端。

```javascript
// Worker腳本 - deepseek-proxy

// DeepSeek API 的目標端點
const DEEPSEEK_API_BASE_URL = "https://api.deepseek.com";

// 處理所有傳入請求
addEventListener("fetch", event => {
  event.respondWith(handleRequest(event.request));
});

async function handleRequest(request) {
  // 檢查請求方法，只處理POST請求（DeepSeek API通常是POST）
  if (request.method !== "POST") {
    return new Response("Method Not Allowed", { status: 405 });
  }

  // 從環境變量中獲取DeepSeek API Key
  // 注意：DEEPSEEK_API_KEY 會在第三步設置
  const deepseekApiKey = DEEPSEEK_API_KEY; 
  if (!deepseekApiKey) {
    return new Response("DeepSeek API Key is not configured.", { status: 500 });
  }

  // 複製原始請求，並修改請求頭
  const newHeaders = new Headers(request.headers);
  newHeaders.set("Host", "api.deepseek.com"); // 設置正確的Host頭
  newHeaders.set("Authorization", `Bearer ${deepseekApiKey}`); // 添加API Key

  // 構建新的請求 (轉發到 DeepSeek API)
  const proxyRequest = new Request(request, {
    headers: newHeaders,
    redirect: "follow", // 處理重定向
  });

  // 從原始請求中提取路徑，並構建目標URL
  // 例如：如果請求是 /v1/chat/completions，則目標是 https://api.deepseek.com/v1/chat/completions
  const url = new URL(request.url);
  const targetUrl = `${DEEPSEEK_API_BASE_URL}${url.pathname}${url.search}`;

  try {
    // 發送請求到 DeepSeek API
    const response = await fetch(targetUrl, proxyRequest);

    // 複製並返回 DeepSeek API 的響應
    // 注意：如果響應是Stream，需要特別處理
    const newResponse = new Response(response.body, response);
    return newResponse;

  } catch (error) {
    console.error("Error proxying request:", error);
    return new Response(`Proxy Error: ${error.message}`, { status: 500 });
  }
}
```

將上述代碼粘貼到您嘅Worker編輯器中，然後點擊「**保存並部署**」。

### 第三步：配置環境變量

為咗安全起見，我哋唔會將API Key直接寫死喺代碼中，而係將佢作為環境變量配置。

1.  喺Worker編輯器頁面，點擊左側導航欄嘅「**設定**」（Settings）。
2.  選擇「**變數**」（Variables）。
3.  喺「**環境變數**」（Environment Variables）部分，點擊「**新增變數**」。
4.  **變數名稱 (Variable name)：** `DEEPSEEK_API_KEY`
5.  **值 (Value)：** 粘貼您嘅DeepSeek API Key。
6.  點擊「**儲存**」。

### 第四步：部署與測試

當您保存並部署咗Worker之後，Cloudflare會為您提供一個臨時嘅Worker URL（例如：`deepseek-proxy.yourusername.workers.dev`）。您可以使用呢個URL嚟測試您嘅中轉站。

**測試方法：**

您可以使用`curl`命令或者Postman等工具進行測試。假設您想調用DeepSeek嘅`v1/chat/completions`接口：

```bash
curl -X POST \
  "https://deepseek-proxy.yourusername.workers.dev/v1/chat/completions" \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [
      {"role": "user", "content": "你好，DeepSeek！請介紹一下你自己。"}
    ],
    "model": "deepseek-chat",
    "stream": false
  }'
```

將`deepseek-proxy.yourusername.workers.dev`替換為您實際嘅Worker URL。如果一切順利，您將會收到來自DeepSeek模型嘅響應。

**自定義域名配置 (可選)：**

如果您有自己嘅域名並已接入Cloudflare，您仲可以將Worker部署到您嘅自定義子域名上：

1.  喺Worker編輯器頁面，點擊「**觸發器**」（Triggers）。
2.  喺「**自訂網域**」（Custom Domains）部分，點擊「**新增自訂網域**」。
3.  輸入您希望使用嘅子域名（例如：`ai.yourdomain.com`），然後按照指示完成配置。Cloudflare會自動創建相關嘅DNS記錄。

## 進階優化：打造更高可用性與效能

完成基本中轉站搭建後，我哋仲可以做更多優化，將佢打造成一個更強大、更穩定嘅服務。

### 引入Cloudflare R2進行響應緩存

對於DeepSeek API嘅某些請求，例如提供固定回應或數據量較小嘅請求，可以考慮使用Cloudflare R2作為緩存層。

1.  **創建R2 Bucket：** 喺Cloudflare儀表板，進入「**R2**」，創建一個新的Bucket。
2.  **修改Worker腳本：**
    *   喺Worker腳本中加入R2嘅綁定（Binding），例如命名為`DEEPSEEK_CACHE`。
    *   編寫緩存邏輯：喺每次發送請求前，先檢查R2中是否有對應嘅緩存數據；如果命中緩存，直接返回；如果無，則向DeepSeek API發送請求，並將響應存儲到R2中以供下次使用。
    *   呢個策略可以大幅減少對DeepSeek API嘅重複請求，降低延遲，甚至可以喺DeepSeek API暫時唔可用時提供舊嘅緩存數據，提高高可用性。

### WAF與限速策略

Cloudflare強大嘅網絡安全功能，可以保護您嘅中轉站免受惡意攻擊同濫用。

*   **WAF規則：** 配置WAF規則，阻擋常見嘅Web攻擊，例如SQL注入、跨站腳本攻擊等。
*   **限速規則：** 喺Cloudflare儀表板嘅「**安全性**」>「**限速**」中設置，對特定URL或IP地址設置請求頻率限制，防止DDoS攻擊或惡意程式調用導致API Key被濫用。
*   **Access應用：** 設置Cloudflare Access，只允許特定用戶或IP範圍存取您嘅中轉站。

### 自定義路由與多區域部署

如果您的應用用戶分佈喺多個地理位置，或者對延遲有極高要求，可以考慮更進階嘅路由策略：

*   **基於地理位置路由：** 利用Cloudflare Worker嘅地理位置信息，將請求路由到距離用戶最近嘅DeepSeek數據中心（如果DeepSeek提供多個數據中心嘅話），或者根據業務邏輯將請求分發到唔同嘅下游服務。
*   **多Worker部署：** 部署多個Worker實例，每個實例服務於特定地區或特定功能，進一步分散風險同提升擴展性。

### 日誌與監控

Cloudflare Workers提供詳盡嘅日誌同分析功能。

*   **Worker日誌：** 定期檢查Worker日誌，了解請求狀態、錯誤信息，以便快速排查問題。
*   **Cloudflare Analytics：** 監測Worker嘅請求量、響應時間、錯誤率等關鍵指標，掌握中轉站嘅運行狀況。
