配置DEEPSEEK_API_KEY = "YOUR_DEEPSEEK_API_KEY"DEEPSEEK_API_URL = "https://api.deepseek.com/chat/completions"# Redis 配置REDIS_HOST = "localhost"REDIS_PORT = 6379REDIS_DB = 0CACHE_EXPIRATION_SECONDS = 3600 json.loads(cached_response)        print("呼叫 DeepSeek API")    headers = {        "Authorization": f"Bearer {DEEPSEEK_API_KEY}",        "Content-Type": "application/json"    }    payload = {        "model": model,        "messages": prompt_messages,        "stream": False    }    try:        response = requests.post(DEEPSEEK_API_URL, headers=headers, json=payload, timeout=30)        response.raise_for_status() # 如果請求失敗則拋出異常        api_response_data = response.json()                # 將 API 響應存儲到 Redis 緩存中        r.setex(cache_key, CACHE_EXPIRATION_SECONDS, json.dumps(api_response_data))        return api_response_data    except requests.exceptions.RequestException as e:        print(f"DeepSeek API 呼叫失敗: {e}")        # 可以在此處添加重試邏輯，參考這篇 [DeepSeek API 緩存命中率怎麼才能更高？想提高緩存命中率，關鍵點就是標準化輸入。比如說，在把用戶輸入當作緩存鍵之前，你可以先把文本轉成小寫，去掉多餘的空格或者標點符號。如果你的應用還用到了向量搜索來找到相關內容，並且把這些內容和用戶問題一起發給 DeepSeek API，那緩存鍵就得把構成最終 `messages` 的所有元素都包含進去，這樣才能保證只有完全一樣的請求才能命中緩存。另外，合理設置緩存過期時間也很重要，別設得太短頻繁失效，也別設得太長導致數據過時。想了解 DeepSeek API 密鑰和限制更多信息，可以參考這篇 [DeepSeek API 金鑰管理與用量限制](/posts/deepseek-api-key-and-limits/)。### Q2: 啥時候需要清掉緩存？緩存清除（cache invalidation）其實是個挺麻煩的話題。通常來說，當你的基礎數據（比如知識庫、RAG 文檔）更新了，或者 DeepSeek 模型本身有大版本更新時，你就得考慮清掉相關緩存了，這樣才能保證用戶拿到的是最新信息。你可以用 Redis 的 `DEL` 命令手動清掉特定的鍵，或者設置一個更短的過期時間讓它自己失效。要是整個應用都更新了，可以考慮用 `FLUSHDB` 或 `FLUSHALL` 命令，但這倆得小心用，因為它們會把所有緩存都清光光。### Q3: 部署 Redis 有沒有什麼坑要注意？在生產環境部署 Redis 的時候，有幾點小建議給你：*   **持久化**：考慮把 Redis 的 RDB 或 AOF 持久化功能開起來，這樣服務重啟也不怕數據丟失了。*   **高可用性**：對於那些併發量超大的應用，可以考慮部署 Redis Sentinel 或 Redis Cluster，實現高可用和負載均衡。*   **安全性**：設個強密碼，別把 Redis 端口直接暴露在公網上，最好在防火牆後面跑。*   **監控**：配上監控工具，實時盯著 Redis 的內存使用、連接數、命中率這些指標，有問題也能及時發現解決。## 總結一下總的來說，給 DeepSeek API 搭建個 Redis 緩存層，咱們不光能有效控制 API
